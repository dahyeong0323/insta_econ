# AGENTS.md

## Product North Star
- 이 프로젝트의 최종 목표는 중학교 경제 개념을 자동으로 선정하고, 카드뉴스를 생성하고, 사람 승인 후 인스타그램까지 업로드하는 자동화 콘텐츠 시스템을 만드는 것이다.
- 핵심 흐름은 `research -> Telegram script approval -> card generation -> Telegram image approval -> Instagram publish` 이다.
- 모든 구현은 이 최종 흐름에 기여하는 방향으로 진행한다.

## Current Product Shape
- 현재 프로젝트는 Next.js 기반 카드뉴스 생성 앱이다.
- 디자인 품질은 이미 중요한 자산이므로 새 기능을 추가할 때 기존 시각 톤과 카드 구성 완성도를 해치지 않는다.
- 기존 생성 파이프라인은 최대한 재사용하고, 새 기능은 파이프라인 바깥에 덧붙이기보다 상태 흐름 안으로 편입한다.
- 현재 활성 운영 배포는 GitHub 연동 Vercel 프로젝트 `insta-econ-fzr1` 기준으로 본다.
- 운영 기준 공개 도메인, webhook URL, 이미지 공개 URL은 항상 현재 활성 Vercel 프로젝트 기준으로 다시 확인한다.

## Primary Users
- 운영자: 텔레그램으로 초안과 이미지를 받고 승인 또는 수정 요청을 내리는 사람
- 최종 독자: 중학생 수준에서 이해 가능한 경제 카드뉴스를 보는 인스타그램 사용자

## Core Content Rules
- 모든 콘텐츠는 `중학교 경제 개념` 기준으로 쉽게 설명한다.
- 어려운 경제 용어는 바로 풀어서 설명한다.
- 카드뉴스는 정보 전달보다도 `짧고 명확한 이해`를 우선한다.
- 자극적인 톤보다 믿을 수 있고 또렷한 톤을 유지한다.
- 사실 근거가 불분명한 내용은 생성하지 않는다.
- 이미 인스타그램에 게시한 주제와 `완전히 같은 내용`은 다시 만들지 않는다.
- 이미 게시한 주제와 `개념적으로 너무 가까운 내용`도 되도록 피한다.
- 새 주제를 고를 때는 항상 과거 게시 이력과 비교해서 중복, 재탕, 과도한 유사성을 먼저 검사한다.

## North Star Workflow
1. Research agent가 4시간마다 주제 하나를 선정한다.
2. 선정된 주제로 짧고 명확한 카드뉴스 대본을 만든다.
3. 대본을 Telegram으로 운영자에게 보낸다.
4. 운영자가 `OK` 하면 현재 카드뉴스 생성 파이프라인으로 제작한다.
5. 생성된 카드뉴스 이미지를 Telegram으로 다시 보낸다.
6. 운영자가 최종 승인하면 Instagram API로 업로드한다.
7. 실패한 단계는 재시도 가능해야 하고, 어떤 단계에서 멈췄는지 항상 추적 가능해야 한다.
8. 게시가 완료되면 주제, 핵심 개념, 캡션, 게시 시각, 인스타 결과값을 DB에 기록하고 이후 연구 단계에서 반드시 참조한다.

## Build Priorities
1. Telegram 승인 흐름 구축
2. 승인된 대본을 기존 카드뉴스 생성 파이프라인과 연결
3. 이미지 렌더 결과를 Telegram으로 전달
4. 최종 승인 후 Instagram 업로드 자동화
5. 게시 이력 DB와 중복/유사 주제 방지 로직 추가
6. 스케줄링, 재시도, 로그, 실패 복구 고도화

## Architecture Direction
- 기존 `src/lib/agents`, `src/lib/runs`, `src/app/api` 구조를 우선 활용한다.
- 가능하면 새 기능도 `run` 단위 상태 머신으로 설계한다.
- 게시 완료 이력은 파일만으로 끝내지 말고, 연구 에이전트가 조회할 수 있는 DB 테이블 또는 동등한 저장소에 남긴다.
- 상태는 최소한 아래 단계를 구분할 수 있어야 한다.
  - `researched`
  - `script_pending_approval`
  - `script_approved`
  - `rendering`
  - `image_pending_approval`
  - `image_approved`
  - `publishing`
  - `published`
  - `failed`
- 단순 성공/실패만 기록하지 말고 `무엇을 보냈는지`, `누가 승인했는지`, `어느 메시지에 대한 응답인지` 추적 가능해야 한다.
- 연구 단계는 과거 게시 이력을 읽고 `주제 중복`, `핵심 개념 중복`, `설명 각도 유사성`을 확인해야 한다.

## Implementation Rules
- 새 기능을 만들 때 기존 생성 로직을 복사하지 말고 재사용 가능한 함수 또는 서비스로 분리한다.
- 텔레그램, 인스타그램, 스케줄러 연동 로직은 UI 코드와 분리한다.
- 외부 API 연동 코드는 얇은 adapter 레이어로 감싼다.
- 승인/게시 같은 운영 이벤트는 나중에 DB로 옮기기 쉽게 설계한다.
- 게시 완료 레코드는 나중이 아니라 실제 운영 전에는 DB로 관리하는 것을 기본 방향으로 본다.
- 지금은 로컬 파일 저장을 써도 되지만, 데이터 구조는 DB 이전이 가능하도록 명확하게 유지한다.
- 한 번에 전체 리팩터링하지 않는다. 항상 작동하는 작은 단계로 나눈다.

## UX and Design Guardrails
- 기존 카드뉴스 디자인 톤을 유지한다.
- 중학생 대상 콘텐츠이므로 한 슬라이드 안에 과도한 정보량을 넣지 않는다.
- 텍스트, 이미지, 강조 요소는 모바일 인스타 소비 환경을 기준으로 판단한다.
- 자동화 기능을 추가해도 수동 검수 경험은 더 편해져야지 복잡해지면 안 된다.

## Suggested Near-Term File Ownership
- `src/lib/agents/*`: 콘텐츠 생성과 QA 파이프라인
- `src/lib/runs/*`: 실행 상태 저장과 단계 진행
- `src/app/api/*`: 외부 트리거와 승인 처리용 엔드포인트
- future `src/lib/integrations/telegram/*`: 텔레그램 송수신
- future `src/lib/integrations/instagram/*`: 인스타 업로드
- future `src/lib/research/*`: 주제 선정과 대본 초안 생성

## Decision Rules For Agents
- 새로운 작업을 시작할 때 항상 이 기능이 최종 워크플로우의 어느 단계에 속하는지 먼저 판단한다.
- 방향이 애매하면 `디자인 추가`보다 `상태 추적 가능성`과 `승인 플로우 안정성`을 우선한다.
- 자동 게시보다 승인 안전장치를 우선한다.
- 인스타 업로드는 마지막 단계에서 붙인다.
- 운영자가 Telegram에서 쉽게 이해하고 승인할 수 있는 메시지 포맷을 우선 설계한다.
- Research agent를 만들 때는 `새로운 주제를 잘 고르는 능력`보다 `이미 올린 주제를 다시 고르지 않는 안전장치`를 먼저 만든다.

## Definition of Good Progress
- 승인 전후 상태가 명확하다.
- 실패 시 복구 지점이 분명하다.
- 운영자가 텔레그램만으로 현재 상황을 이해할 수 있다.
- 카드뉴스 품질이 기존보다 떨어지지 않는다.
- 이후 DB, 큐, 배포 환경으로 확장하기 쉬운 구조다.
- 이미 게시된 콘텐츠와 겹치지 않는다는 근거를 시스템이 설명할 수 있다.

## Commands

- 빌드: `npm run build`

## Working Agreement For Codex
- 코딩 전에 현재 구조를 먼저 읽고, 기존 파이프라인을 재사용할 수 있는지 확인한다.
- 큰 목표를 잃지 않도록 작업마다 이 문서의 `North Star Workflow`와 `Build Priorities`를 다시 맞춘다.
- 임시 구현을 하더라도 나중에 Telegram 승인 흐름과 Instagram 업로드 흐름으로 자연스럽게 이어질 수 있게 만든다.
- 작업을 시작하기 전에 `STATUS.md`를 확인해서 현재 진행 상황과 다음 우선순위를 파악한다.
- 구현이 끝난 항목은 즉시 `STATUS.md`에서 완료로 바꾸고, 다음에 할 일을 함께 갱신한다.
- 코드나 운영 문서를 바꿨다면 관련 검증을 먼저 돌리고, 가능하면 같은 턴에 `git push`까지 완료해서 GitHub 연동 Vercel이 자동 재배포되도록 마무리한다.
- 자동 배포가 붙는 작업은 푸시 후 실제 Vercel 배포가 최신 커밋을 반영했는지 확인하고 닫는다.
